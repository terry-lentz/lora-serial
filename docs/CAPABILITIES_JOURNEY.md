# Capabilities & engineering journey

What this link can do, the problems we hit getting there, and how each was solved.
Together these are the features that make a **lean, reliable, fast point-to-point
"LoRa serial cable"** — and where, for the *P2P* job specifically, it stands up to
(and in places exceeds) general stacks like Reticulum. See
[SW_ALTERNATIVES.md](./SW_ALTERNATIVES.md) for the honest head-to-head, [DESIGN.md](./DESIGN.md)
for the transport spec, [RESEARCH.md](./RESEARCH.md) for the techniques, and
[SECURITY.md](./SECURITY.md) for the crypto plan.

## Capabilities (and the problem each solves)

### Reliable transport
| Capability | What it solves |
|---|---|
| **Selective-Repeat ARQ + SACK bitmap** | Resends only the frames actually lost (not the whole window like Go-Back-N) → far less wasted airtime under loss. |
| **BDP-sized window (per mode)** | Keeps the pipe full over the bandwidth-delay product → fixed the multi-window "stall at one window" on full-size frames. |
| **Out-of-order reassembly** | Buffers frames that arrive out of order and delivers in order (GBN discarded them). |
| **Signaled burst hand-off** (`F_MORE` + per-frame deadline reset) | A burst of *any* length completes — no fixed turn-timer cutting it off mid-burst. |
| **Receiver backpressure** | When the receive ring is full, the frame is held & re-acked (never silently dropped); delivers when the app drains. |
| **Epoch resync on reboot** | A rebooted peer resyncs sequence numbers automatically (no manual reset). |
| **Session reset on host reconnect** | A new host session never inherits the previous one's stale queued/in-flight bytes. |

### Host (USB) link
| Capability | What it solves |
|---|---|
| **TinyUSB OTG-CDC stack** | Real USB flow control (NAK) instead of the ESP32-S3 USB-Serial/JTAG silently dropping bytes on bursts (the original ~270 B/s ceiling). |
| **Checksummed USB framing + length-cap resync** | Detects any corruption/desync on the USB hop and re-syncs within one frame — never delivers garbage. |
| **PSRAM ingest ring + bulk USB reads** | A fast `cat bigfile` floods USB (~1 MB/s; the "115200 baud" is cosmetic) far faster than the ~2 KB/s link. The arduino-esp32 USB-CDC core silently drops on overflow (no NAK — upstream bug [#10836](https://github.com/espressif/arduino-esp32/issues/10836)). We bulk-drain the FIFO into a 2 MB PSRAM ring → every realistic transfer is absorbed losslessly with **no client-side flow control**. Verified: 128 KB fast-overrun transfer byte-exact (`idrop=0`). |
| **Always-drain `usbPoll` + 2 KB staging** | Never stalls the USB read, so the FIFO doesn't overflow; backpressure lives downstream. |
| **Non-blocking host TX ring** | A slow terminal can't block the radio loop (a blocking write used to wedge the whole link). |
| **XON/XOFF flow control** + **tiny bulk pacing** | Keeps the host from overrunning the link ring / USB FIFO at the fastest modes. |
| **No-reset serial open** | Opening the port doesn't toggle DTR/RTS into the bootloader (which bricked the link). |
| **Reconnect output flush** | Reconnecting a terminal doesn't get the previous session's buffered output dumped at it. |
| **USB-hop diagnostics** (`ugood`/`udrop` telemetry) | Makes host-hop byte loss measurable — turned guesswork into root-cause. |

### Radio / RF
| Capability | What it solves |
|---|---|
| **Per-mode ToA-derived timing** | Turn timing auto-computed from `getTimeOnAir()` → any SF/BW is tuned automatically, no hand-constants. |
| **Auto TX power control** (symmetric-link heuristic) | Backs power off when the link is over-strong (front-end saturation at close range) and ramps up when weak — never above the configured legal max. Foundation for ADR. |
| **Configurable SF/BW/CR/power + named modes** | turbo / fast / medium / slow / far — speed-vs-range presets over the standard knobs. |
| **Half-duplex turn-taking MAC** | Coordinates a single shared radio for two peers with minimal dead time. |

### Security & integrity
| Capability | What it solves |
|---|---|
| **Ascon-128 AEAD encryption** (pre-shared key) | Confidentiality **and integrity**: the NIST-lightweight-standard cipher (validated against the official KAT vectors). Every frame — header included — is authenticated, so tampering/bit-flips are detected and dropped. |
| **64-bit nonce counter persisted in NVS** | No keystream reuse across reboots: the counter resumes past its saved value (skips unsaved counters), the exact LoRaWAN-1.1 fix for the counter-reset bug. |
| **Sliding-window anti-replay** | Recorded ciphertext frames can't be re-injected within a session (64-frame window tolerates reordering). Plaintext frames are rejected on an encrypted link. |
| **Link watchdog** | Tears down the login session when the air link drops, so a reconnect must re-authenticate (no resumed authenticated shell). |
| **LoRa hardware CRC + per-frame integrity** | Corrupt air frames are dropped and retransmitted (never delivered). |

### Compression & data
| Capability | What it solves |
|---|---|
| **Per-frame heatshrink compression** (only if it shrinks) | Cuts airtime on compressible data (text/logs ~2× the raw rate); never expands incompressible data. |
| **Byte-exact transparent pipe** | Arbitrary binary passes intact end-to-end (verified with random payloads). |
| **Pure transparent byte pipe** | The link stays a clean byte pipe; any terminal cooking (pty/line discipline) is left to the OS at each end — proper layering. |

### Usability
| Capability | What it solves |
|---|---|
| **Client = any plain serial terminal** | No special app on the roaming end — tio, PuTTY, a phone serial app all work. |
| **AT/`+++` command mode** (Hayes-style) | Configure the radio (power/address) and query live link state (`AT+LINK?` → RSSI/SNR/power) out-of-band from the *same* plain terminal — no host software, no extra wires. Covers the link-state role DCD can't (TinyUSB CDC has no device→host carrier API). |
| **Unified firmware** (role from address) | One binary for both ends; the lower address auto-initiates → boards are interchangeable, no per-board build. |
| **OS-native login over LoRa** (`agetty`) | Because the link is a transparent serial port, you point `agetty`/`serial-getty` straight at it for a real PAM login + shell — no project-specific server. systemd unit + udev rule in `deploy/`. |
| **Native simulation test suite (26 tests)** | Loss, reordering, backpressure, compression edges, reconnect resync, SR efficiency, AEAD KAT/replay, X25519 pairing — caught real bugs before hardware. |
| **Throughput harness + flashing helper + udev rules** | Reproducible benchmarking; button-free flashing over native USB. |

## Problems we hit, and how we solved them (the journey)
A condensed engineering log — each was a real bug or wall, and the fix became a feature above.

1. **ISR crash under load** — `digitalWrite` in an IRAM ISR crashed the chip → moved the LED out of the ISR.
2. **Decompress hang** — heatshrink could loop forever → output-bound guards.
3. **Bulk stall at ~240–850 B** — the USB-Serial/JTAG stack dropped host bytes on bursts → first paced, ultimately **switched to TinyUSB** (real flow control). ~270 → 3000+ B/s.
4. **Compression corrupted incompressible data** — a `c>255` length truncation; the sim missed it because tests only used *compressible* payloads → fixed + added an incompressible test.
5. **Login froze after password** — agetty's auto-baud mangled input → spawn `login(1)` directly.
6. **Reconnect desync / resumed sessions** — epoch handshake + a link watchdog that logs out on link loss.
7. **The phone-terminal saga** — chased telnet IAC and CR/LF translation; the real cause was the USB stack. Removed the telnet/CR cruft (the pty handles CR; the byte ferry stays transparent).
8. **The multi-window stall on full frames** — Go-Back-N + a tiny window + rigid turn timers → **Selective-Repeat + SACK + BDP window + per-frame burst deadline**. Bulk binary became reliable.
9. **USB framing desync** — a single dropped byte corrupted the stream forever → **checksum + length-cap resync**.
10. **Cross-run stale data / dirty reconnects** — the board kept link state across host sessions → **session reset on reconnect**.
11. **RX overload vs USB-hop drops** — strong-signal corruption *looked* like RF overload; `udrop` telemetry proved it was the **USB hop at high feed rate** (fixed with a tiny pace) while **auto TX power** handles real saturation.
12. **Crypto was confidentiality-only** — AES-CTR had no integrity (bit-flipping), reused keystream on reboot (counter reset to 0), and no replay protection. Replaced with **Ascon-128 AEAD** (KAT-validated) + **NVS-persisted 64-bit counter** (reboot-safe nonce) + **anti-replay window**, header authenticated. Tag length is a build flag (8/16).
13. **Large transfers dropped/stalled at ~30–50 KB** — *not* the radio or transport (both proven byte-exact at 256 KB / 1 MB in sim, and at 128 KB on hardware once paced). A per-frame byte counter (`hin`) pinpointed the loss to **host→board USB ingestion**: the arduino-esp32 USB-CDC core drains the FIFO into a small queue and drops on overflow (no NAK — upstream [#10836](https://github.com/espressif/arduino-esp32/issues/10836)) when a fast `cat` outruns the link. Fixed with a **2 MB PSRAM ingest ring + bulk reads**; 128 KB fast-overrun now byte-exact with zero drops.
14. **Turbo was at ~half its PHY ceiling** — profiling showed only ~24% airtime used, so turbo was **turnaround/overhead-bound, not airtime-bound**. Two fixes ~**doubled it to ~8 KB/s** (≈ 56K modem): a **BDP-sized per-mode window** (16 frames/burst at turbo so one turn-around amortizes over a long burst; auto-scaled from ToA, down to 2 at SF12), and **skipping the 30 ms interactive-echo piggyback wait on one-way bulk** (pure dead-time in the ACK path). Reliability unchanged (128 KB overrun + 24/24 sim still byte-exact).
15. **Auto-power starved the link dead (the symmetric-link assumption)** — a responder went deaf mid-run (`rssi` pinned, `snr` garbage, `tx=0`) while the initiator hit ~100% retx with its power floored at `pwr=-10`. Root cause: `AdjustTxPower()` set *our* power from the RSSI *we* receive, assuming a symmetric link. On an asymmetric path (different loss each way + a noisy close-range bench) the side that hears its peer strongly floors its own power, which drops the peer below its demod SNR — a stable, dead equilibrium. Modeled it in the sim (`test_autopower_own_rssi_starves_asym`) and it **reproduced the exact hardware numbers** (initiator floors to -10, responder can't demod, initiator still hears the responder). Fix: **auto-power is now opt-in (`FEAT_APWR`, `AT+APWR`), OFF by default** — fixed power keeps an asymmetric link up (sim-validated); a correct peer-SNR-feedback loop (also sim-validated) is future work. This restored the dead link to byte-exact.
16. **No recovery for a wedged radio** — a coredump confirmed the responder had panicked, and there was no firmware path to recover a radio whose RX silently wedges (rendezvous only changes *mode*, and only in `auto`). Added a **radio-stuck watchdog**: on prolonged no-valid-RX it does a full `begin()` re-init at the *current* mode (mode- and link-preserving, runs in all modes), layered below rendezvous. Validated in sim first (`test_sim_radio_stuck_*`, incl. a teeth test that the link stays dead without it). Caveat found on hardware: a full **reboot** clears a wedge that `begin()` alone does not — a future watchdog should `reset()` the chip (toggle NRST) for a deeper recovery.
17. **"Encryption broke the speed?" — ruled out by isolation** — fast modes ran below the README numbers with ~19–33% retx *at +20 dB SNR* (loss that isn't SNR-driven). Hypothesis: the per-frame Ascon decrypt. Tested directly with `AT+ENC=0` on both ends: throughput was **unchanged** (ENC off ≈ ENC on) — decrypt is ~tens of µs on a 240 MHz S3, far too fast to matter. The real cause is the **RX re-arm deaf window**: the receive path did `startReceive → readData → standby` *per frame* and re-armed on the next call, so a back-to-back frame arriving during inter-frame processing was missed (research confirms this is a known RadioLib/SX126x anti-pattern — the SX126x stays in RX after a packet). Modeled it in the sim (`test_phys_rx_rearm_miss_collapses`: continuous-RX ~2.3× faster than per-frame standby). Implemented continuous RX (arm once, no per-frame standby) behind a build flag **`RX_CONTINUOUS` (default OFF)**. Along the way found a real bug: `startTransmit()` assumes the radio is in standby (RadioLib's blocking `transmit()` standbys first), so leaving continuous RX requires an explicit `standby()` before TX or the transmit is malformed — fixed. It is **OFF by default and should not be enabled**: besides being unvalidatable on a bimodal bench (a degraded window made the known-good per-frame build fail identically), the continuous-RX build **HUNG the responder hard** — zero LED activity, unresponsive to AT, **no panic/coredump, and the 25 s software watchdog did not recover it** (only a reflash did), a failure mode never seen on the per-frame build. So it's two problems: a continuous-RX trigger *and* a more general **watchdog gap** (a hung loop must always self-recover, and this one defeated the watchdog — likely a hang with interrupts/timer blocked). One good-window run did show turbo +25% (3.95 vs 3.17 KB/s), so the *direction* is right, but it must not ship until the hang is root-caused and the watchdog made to catch it. (`getRxBufferStatus` is *not* the issue — RadioLib `readData()` already reads at the rolling offset; the real bug found was that `startTransmit` assumes STANDBY, so leaving continuous RX needs an explicit `standby()` before TX.)
19. **A hung loop defeated the software watchdog (robustness gap) — fixed.** A deeply hung firmware state showed no LED, no AT, no panic, and the 25 s `esp_timer` watchdog never rebooted it. The fix: put the loop task under the IDF **Task Watchdog** (`enableLoopWDT()`; the framework has `CONFIG_ESP_TASK_WDT_PANIC=y`, so a stalled loop now reboots), plus a software **no-progress reboot** (`MaybeReboot`/`DiagRebootNoProgress`, `lastreset=NO-PROGRESS`) for a deaf-but-alive wedge, and an **NRST hardware radio reset** (`radio.reset()`) in the radio-stuck recovery. Recovery is now layered: radio re-init → rendezvous → NRST reset → reboot → TWDT. Validated: boards self-recover instead of needing a reflash.
20. **Continuous RX vs the SX1262 AGC lockup — and a wrong turn corrected.** The throughput ceiling is the per-frame RX **re-arm deaf window** (~20-40 % retx at high SNR). Early continuous-RX attempts (arm-once `RX_CONTINUOUS`, then an **interrupt-driven** radio task) showed the responder go deaf (`snr~-27`) where per-frame stayed healthy (snr ~12), which pointed at the "AGC lockup in RX" errata (per-frame's standby-between-frames resets the AGC). The fix was *thought* to be `restartAGC()` "in a newer RadioLib." **Both halves of that were wrong** (corrected 2026-06-30): 7.7.1 *is* the latest RadioLib, and the method is `resetAGC()` and already present — no upgrade needed. See entry 22 for what `resetAGC()` actually did.
21. **Close-range desense — lower the power.** Much of the session's "deafness/wedge" at the cm bench was **front-end desense** from +22 dBm transmitters centimetres apart. Measured (per-frame medium): 22 dBm → 0.39 KB/s / 44 % retx; 10 dBm → 0.48 / 23 %; **2 dBm → 0.69 / 28 %**. So fixed-max power is wrong at close range; reducing it recovers the link. The proper fix is a correct **peer-SNR-feedback auto-power** (#53) that lowers power when the peer hears us strongly — which also cures the earlier asymmetric-starvation. (Neither Meshtastic nor Reticulum auto-adjusts power; this is our own feature.)
22. **`resetAGC()` WEDGES the SX1262 — the real cause of the "responder hangs" saga.** To prevent the AGC lockup (entry 20) I added RadioLib's `resetAGC()` (warm `sleep(true)` → recalibrate) to the continuous-RX path. After that build ran, **every** subsequent reflash — including the known-good **per-frame** build — left the responder dark (no LED/AT/coredump). Root-caused with no JTAG by adding a retry loop that prints the error: `radio.begin()` returns **err -2 `CHIP_NOT_FOUND`**, and even an aggressive NRST (3×100 ms) won't clear it — **only a VBUS power-cycle does.** So `resetAGC()`'s `sleep()` left the chip unreachable over SPI, and the old `setup()` `while(true){delay(500)}` dead-radio trap then *looked* like an MCU hang on every boot. This also means the earlier continuous-RX/INTR_RX "deaf/hang" results were partly against an already-wedged radio. Fixes: (a) **removed `resetAGC()`** — AGC reset uses a plain `standby()`; (b) reworked continuous RX to a **single-threaded main-loop** design (`RX_CONTINUOUS`); (c) replaced the silent dead-radio trap with `RadioHardNrst()` + a **retry/self-heal** bring-up so a wedged radio is visible and recovers on power-cycle instead of hanging dark; (d) added `-DDEBUG_BRINGUP=1` staged serial markers for no-JTAG bring-up debugging (lesson: native-USB CDC drops boot-time prints with no DTR, so only *continuously* printed markers are visible). Lesson: **never warm-`sleep()` this SX1262 in the hot path**, and never let `setup()` hang silently on a radio failure.
23. **The real villain all along: a loose board-to-board connector.** After the `resetAGC` wedge (entry 22) survived even a VBUS power-cycle, RadioLib debug showed the SX1262 version register reading a constant mostly-`0xFF` (`ff ff ff fe 0b …`) — the chip wasn't driving MISO. Not a software wedge: a **marginal B2B connector** on one XIAO/Wio-SX1262 unit (pressing/wiggling it made the LED resume). Swapping in another board fixed it instantly. **This intermittent connection — not firmware — was behind most of the multi-session "responder goes deaf / wedges / hangs" saga.** On a *solid* link the truth is simple: the **per-frame path is already 0 % retx** at fast modes (the earlier 42–58 % retx was the flaky connector + close-range desense), and **interrupt-driven RX (`INTR_RX`) is byte-exact across every mode** with no hang/deafness. Continuous RX gives ~no throughput gain over per-frame on a good link (both 0 % retx) — its whole rationale (the re-arm deaf window) was really the hardware. Lessons baked in: a self-healing radio bring-up (entry 22c), and — to make the *next* loose-connector instantly obvious — moving device identity + the USB descriptor to the chip MAC so the `/dev/serial/by-id` name self-documents (planned). Throughput finally measured clean (INTR_RX, byte-exact, both compression cases) — see THROUGHPUT.md.
24. **Slow modes need ToA-scaled recovery timeouts.** With a solid link, `slow` (SF9) worked but `far` (SF12) reboot-looped: both ends hit `lastreset=NO-PROGRESS`. The recovery watchdogs (radio-stuck reinit 5 s, rendezvous 9 s, no-progress reboot 45 s) were **fixed** while SF12 airtime is ~4× slower, so the 5 s radio-stuck reinit fired *mid-handshake* (a far turn is ~8 s), disrupting the link until the 45 s reboot. Fix: each timeout is now `max(floor, mult × g_retransmit_ms)` — the ARQ retransmit timeout already scales with time-on-air — so the watchdogs stay eager on fast modes (the proven 5/9/45 s floors) and patient on slow ones, preserving the order stuck < rendezvous < reboot. (Also fixed a real `INTR_RX` bug found here: the radio task's idle re-arm timeout aborted slow/far frames whose ToA exceeded it — it now blocks on `portMAX_DELAY` and re-arms only after `RX_DONE`.)
25. **Channels: investigated and removed.** We briefly shipped a global numbered-channel scheme (`AT+CHAN`) — one integer grid laid over the SX1262's whole 150–960 MHz range, `freq = 150 + chan × bandwidth`. In use it was a poor fit: the numbers were huge and unintuitive (US 915 MHz ≈ channel **6120** at BW125), and worse, **bandwidth-dependent** — because the grid step *is* the current bandwidth, the same frequency is a *different* channel at every mode, so a channel number meant nothing without also stating the mode. Channels only really earn their keep as a small, fixed, **per-region** plan (LoRaWAN-style EU868 ch0–7, US915 ch0–63). Rather than keep a confusing universal grid, we **removed `AT+CHAN`/`AT+CHAN?`** and made `AT+FREQ` (plain MHz, mode-independent) the single way to set the carrier — folding the old grid's range check into `AT+FREQ` (rejects outside ~150–960 MHz) so nothing was lost. Region-aware *named* channel plans may return alongside the FHSS work on the [roadmap](./ROADMAP.md), where a shared channel list is the natural foundation.
26. **far/SF12 delivered nothing — a TX-safety timeout shorter than the airtime.** On a solid link, every mode worked except `far`: data was ingested but the responder's `hout` stayed 0 with ~75% retx and a radio-stuck `reinit`, identically at `pwr=22` and `pwr=2` (so *not* desense — ruling that out at the 1.5 m bench took a low-power re-run). Root cause: `TxFrame()`'s stuck-radio safety timeout was a **fixed 8 s**, but a far/SF12/BW125/CR4-8 frame's time-on-air is **~13 s** — so the TX wait broke early and `finishTransmit()` ran on a **truncated frame** the peer could never decode. The comment literally said "Longer than any real ToA," true until SF12 was added. Fixes (sim-first): (a) the safety timeout now **scales with the frame ToA** (`2×g_toa_ms + 1 s`, floored at 8 s) so it always outlasts the airtime — safe because `DiagPet()` (the RX idle hook) feeds both watchdogs throughout the long wait (`test_tx_safety_exceeds_toa_every_mode`); (b) the ARQ retransmit timer became **window-aware** (`(window+2)×ToA×1.4`, was a fixed `12×ToA` ≈ 108 s at far) so a lost tail/lone frame — which the SACK fast-retransmit can't see — recovers in ~one round-trip, not ~2 minutes (`test_far_tail_loss_recovers_promptly`, overhead 14.2→6.1 ToA); (c) the `AT+SPEEDTEST` deadline now **scales its per-byte budget by ToA** (the old fixed 20 s + 4 ms/byte falsely reported a far TIMEOUT). Result on hardware: far now transfers **byte-exact at 0 % retx** (1 KB delivered, responder `hout=1024`). Lesson: every fixed timeout must be checked against SF12's seconds-long airtime — the same class of bug as entry 24's recovery watchdogs.
27. **First-boot proximity pairing (zero-touch unique keys).** MAC role election already made the two boards interchangeable, and `AT+TRAIN` already derived a unique per-pair key over X25519 — but training was a manual step, and without it a pair fell back to the shared built-in key (no per-pair secrecy). Proximity pairing closes that: a board with **no key in NVS** boots straight into pairing — drops to **low TX power** so it reaches a *nearby* board (proximity = human intent, like NFC/BLE), discovers + elects roles, auto-runs the `AT+TRAIN` exchange, **persists role + key**, then jumps to full power; every later boot loads that identity and **skips discovery**. Surfaced for the operator: a pairing LED blink, a periodic `[LoRa-Serial] PAIRING …` line on the USB port, and `ATI state=pairing`. Re-pair with `AT+PAIR` (on demand) or `AT&F` (wipe → re-pair next boot). Built behind `PROX_PAIR` (default ON after validation; `-D PROX_PAIR=0` restores re-elect-every-boot). Hardware-validated on two boards: boot-pairing from an empty NVS (fresh key, both ends matching, persisted), `AT+PAIR` re-pair, paired-boot-skip, and a working data link after pairing — all confirmed. **Known gap:** low power alone is a weak proximity gate — ~0 dBm still linked cleanly at 1.5 m on the bench (SNR 13), so adjacency isn't strictly enforced yet; an RSSI gate (accept a pairing beacon only above, say, −45 dBm — −30 at 5 cm vs −56 at 1.5 m cleanly separates them) is the planned hardening.
28. **Full-auto 1 MB wedge — a watchdog-starvation bug (fixed) layered over an SX1262 mode-switch deaf-lockup (characterized).** With ADR (`auto`), peer-SNR auto-power, and the GFSK rung all on, a 1 MB transfer reboot-wedged at ~18 s every time, killing the host transfer (USB drop). The breakthrough was a **non-perturbing RTC breadcrumb**: each blocking radio op the loop runs stamps `g_dbg_stage`, the RX task stamps `g_dbg_rx`, both `RTC_NOINIT` bytes snapshotted in the IDF Task-WDT ISR *and* the no-progress path so they survive the reboot, reported by `AT+DIAG` as `wedgeop`/`rxop`. (Earlier *serial* markers Heisenbug-shifted the hard wedge into a soft one — a byte write to RTC doesn't perturb timing.) It exposed two independent problems. **Layer 1 — watchdog starvation during recovery (FIXED).** When the link went deaf, the recovery path itself ran >5 s without feeding the 5 s task watchdog, so *recovery tripped the very watchdog meant to let it run* → instant reboot that dropped the transfer. The unfed stretches: `Radio::Reinit()`'s aggressive NRST (~0.9 s of `delay()`) + `begin()` (each SPI op can burn the full BUSY timeout on a wedged chip); the `Rx()`/`Tx()` wait loops whenever the host-I/O idle hook was disabled (pairing NULLs it) or `Rx()` returned at once on a ready-frame burst (it fed only *after* the `Pop`); and `Host::Poll()`'s USB-ingest / `IngestToLink` loops under a host flood. Fix: feed `feedLoopWDT()` through all of them. Result: the task-WDT reboot is **gone** — the transfer now survives the ADR climb (~60 s vs 18 s) and only the *designed* no-progress recovery fires. **Layer 2 — the SX1262 deaf-lockup on a switch under load (CHARACTERIZED).** With the watchdog fed, both ends now reach `lastreset=NO-PROGRESS, wedgeop=i` (in `Reinit`), `rxop=.` (RX task idle): the loop is *alive and recovering*, but an ADR mode switch under sustained bulk intermittently drives the radio into a deep deaf state that runtime recovery — aggressive-NRST `Reinit` **and** rendezvous-to-medium — does **not** clear; only a full MCU reboot does (consistent with the SX126x AGC/state errata, entries 20/22). It is **not turbo-specific** (capping the auto-climb at `fast` still deafed at ~58 s) — it's the *switch under load* itself. By contrast a **fixed** mode is rock-solid for bulk: fixed turbo carried 1 MB **byte-exact at 3.91 KB/s with no reboot** (261 s, past the deaf window), 256 KB earlier the same. Real bugs fixed en route: auto-power *fights* ADR (it floors power for the current slow mode, then ADR climbs into a power-hungry one the now-too-low power can't sustain → power was traced to −6 dBm right before a deaf switch; auto-power is now disabled under ADR); the runtime `Reinit` now uses the boot bring-up's aggressive NRST (a single `radio.reset()` couldn't clear the deep wedge); the RadioLib **SPI BUSY timeout was bounded 1000 → 100 ms** (an unbounded op was a separate starvation vector) and the heap-`new`'d `Module` became a named global to reach it (rule 5); a high-priority RX-task **storm-breaker/yield** (so a noise-driven `RX_DONE` flood can't starve the loop); and the GFSK rung **goes deaf on the switch into it** (`wedgeop=f`) so it stays manual-only in auto. **How to use it:** pin a fixed mode (`AT+FMODE=turbo`/`fast`) for reliable sustained bulk — the stable default path; ADR (`auto`) adapts the mode to changing range/SNR and suits interactive/intermittent use, but ADR + the GFSK rung remain **opt-in/experimental for sustained bulk** pending a fix for the switch-under-load deaf-lockup (the open lead: a recovery that re-inits the SPI peripheral + radio task *without* a full reboot, since only the MCU reset currently clears it). The `wedgeop`/`rxop` breadcrumb stays as a permanent `AT+DIAG` diagnostic.

29. **GFSK `ludicrous` went deaf — the entry-28 storm-breaker throttle broke it (fixed).** After the interrupt-RX/OOP refactor, switching to `ludicrous` and sending data delivered **nothing** and reboot-wedged both boards — even a *manual* `AT+FMODE=ludicrous` on both ends, so it wasn't the mode-switch handshake. LoRa was flawless, which localized the fault to the GFSK-specific path. The chase ruled out — by hardware measurement + transcript forensics — everything "obvious": the GFSK PHY config (bitrate/dev/RxBw/preamble/sync/CRC) was **byte-identical** to the last byte-exact `ludicrous` build, RadioLib was pinned unchanged, frequency/sync matched, and the low-level radio worked. Temporary `AT+LINK?` counters proved DIO1 **fires** for FSK (`rxd` climbs), `readData` **succeeds** (`rxf≈0`), and `startTransmit` **succeeds** (`txa==txo`) — yet the received frames had **garbage lengths** (`getPacketLength` returned 0/178, only rarely a real 26): the receiver was demodulating noise and missing real frames. A forensic diff of the `RadioTask` drain loop against the last known-good checkpoint (which had hit `ludicrous 64 KB @ 9.83 KB/s byte-exact`) found the single regression: the **storm-breaker added in entry 28** appended a `vTaskDelay(1)` **after every processed RX_DONE**. Harmless to LoRa (RX_DONE is sparse; the tick is absorbed in the inter-frame gap), but **fatal to GFSK** — its no-processing-gain receiver fires RX_DONE almost every wake, so the per-frame yield inserted a scheduling gap *between back-to-back burst frames*; the task could no longer drain + re-arm ahead of the next frame, so real frames were dropped/misframed → deaf → NO-PROGRESS reboot. The code was mistaking ordinary GFSK noise for a wedged radio. **Fix:** keep the FIFO-drain hot path **delay-free** (the proven design), and confine loop-starvation protection to a *genuine* deaf flood — a successful frame resets the fail counter, so ordinary GFSK noise (which mostly passes CRC) never trips it; only many *consecutive* failed reads briefly halt RX for the loop's recovery window. **Results — the same throttle broke *both* fast back-to-back-burst rungs (GFSK `ludicrous` AND LoRa `turbo`), so removing it fixed both:** **ADR (`AT+MODE=auto`) is fixed** — it now climbs medium→turbo cleanly and carries **128 KB sustained at 4.83 KB/s, 0 % retx, zero reboots** (26 s, decisively past the ~17 s wall). That means **entry-28's "SX1262 deaf-lock under sustained bulk" was NOT a hardware errata — it was this very throttle**, which had been added *while chasing entry 28*; the "fix" was the bug. **GFSK `ludicrous` is fixed in code** and carried 8 KB byte-exact at 5.91 KB/s when the RF was favorable — but the **cm-range bench is bimodal for GFSK** (entry 18): at a slightly worse board position the demod yields garbage frames, an RF/bench limit (not the code) that needs field-range validation. Lesson (cf. entry 28): a throttle or timeout that is invisible on LoRa's chirp-correlated PHY can be lethal on the fast, sensitive, back-to-back-burst PHYs (GFSK *and* `turbo`) — never put a per-frame delay in the RX drain path.

30. **Auto-power re-enabled under ADR — the "conflict" was the entry-29 throttle; what's left is a power/throughput *tradeoff*, not a wedge.** Entry 28 had *disabled* peer-SNR auto-power whenever ADR was on, on the theory that the two fight: auto-power floors TX power for the current slow mode, then ADR climbs to a power-hungrier one the now-too-low power can't sustain → deaf on the switch (power was traced to −6 dBm right before a deaf climb). But that "deaf" was mostly the storm-breaker throttle (entry 29); with the throttle gone, the pair was worth re-testing together. **They coexist cleanly.** Full `auto` + the GFSK rung + auto-power carried **1 MB end-to-end at 4.83 KB/s, 0 % retx, zero reboots** (212 s continuous), with auto-power actively trimming TX power to **−10 dBm** at 1.5 m — no deaf, no wedge. The old hard conflict was an artifact of the throttle bug, so auto-power now runs under ADR by default (a self-tuning encrypted link out of the box). **What remains is a genuine policy tradeoff, not a failure:** auto-power *minimizes* power, and at close range −10 dBm leaves ADR too little SNR headroom to climb to `ludicrous`, so ADR settles on `turbo` — the same bench that reached ~10 KB/s on `ludicrous` with auto-power *off* runs ~4.8 KB/s with it *on*. Both are lossless; auto-power is simply optimizing for power, not throughput. The cost appears only up close (far away the signal is weak, so auto-power won't cut power and there is nothing to give up) and it protects the RX from close-range saturation. The open follow-up is **coordinated headroom** — have auto-power target the current mode's demod floor *plus a rung of margin* so ADR can still climb, winning both low power and full throughput (to be modelled in the native sim first).

18. **Throughput is compression- and range-sensitive, not broken** — the README's headline rates are **compressible** data; with compression working (`medium` zeros measured **2.65 KB/s, above the 2.0 README**), random data is the worst case and PHY-limited (`medium` ≈ 1.1 KB/s raw). Also confirmed the **cm-range bench is bimodal/wedgy** — the same mode swings 0 ↔ ~3 KB/s run to run because point-blank range causes front-end saturation + local EMI (the laptop/USB next to the initiator's antenna read a ~-32 to -48 dBm noise floor vs the responder's ~-69). A reboot clears wedges. Real headline numbers need **field-range** testing, not a desk bench.

## Where this stands vs Reticulum — for point-to-point
We are **not** a mesh and don't try to be. For the narrow *two-node reliable serial*
job, the specialization wins on several axes:

- **Leaner MAC:** deterministic `F_MORE`/burst hand-off vs CSMA carrier-sense + backoff (pure overhead with only two nodes).
- **No networking tax:** no addresses, routes, or per-packet routing headers.
- **Link-tuned reliability:** SACK + a window sized to *this* link's bandwidth-delay product, timing derived from its exact time-on-air.
- **Auto power control** built in (overload protection + energy), without a network server.
- **A transparent serial *cable*** — plug any program or terminal into either end — rather than a network you build apps into.

Honest counterweight: Reticulum is more **mature**, does **mesh / multi-transport** (out of scope here), and has **forward secrecy** built in. We match it on **authenticated encryption** (Ascon-128 AEAD + replay protection); our forward secrecy (per-session ephemeral X25519 — Fix D) is **built but opt-in/experimental** (the on-air handshake still needs hardening). Same SX1262 PHY → same absolute ceiling; we get *closer to it for the 2-node case* and stay far simpler.

**So: for "a rock-solid, fast, transparent LoRa serial link between two points," this is now a genuinely competitive — and in places leaner/faster — alternative.** For a secure mesh, use Reticulum.
