# Capabilities & engineering journey

What this link can do, the problems we hit getting there, and how each was solved.
Together these are the features that make a **lean, reliable, fast point-to-point
"LoRa serial cable"** ‚Äî and where, for the *P2P* job specifically, it stands up to
(and in places exceeds) general stacks like Reticulum. See
[SW_ALTERNATIVES.md](./SW_ALTERNATIVES.md) for the honest head-to-head, [DESIGN.md](./DESIGN.md)
for the transport spec, [RESEARCH.md](./RESEARCH.md) for the techniques, and
[SECURITY.md](./SECURITY.md) for the crypto plan.

## Capabilities (and the problem each solves)

### Reliable transport
| Capability | What it solves |
|---|---|
| **Selective-Repeat ARQ + SACK bitmap** | Resends only the frames actually lost (not the whole window like Go-Back-N) ‚Üí far less wasted airtime under loss. |
| **BDP-sized window (per mode)** | Keeps the pipe full over the bandwidth-delay product ‚Üí fixed the multi-window "stall at one window" on full-size frames. |
| **Out-of-order reassembly** | Buffers frames that arrive out of order and delivers in order (GBN discarded them). |
| **Signaled burst hand-off** (`F_MORE` + per-frame deadline reset) | A burst of *any* length completes ‚Äî no fixed turn-timer cutting it off mid-burst. |
| **Receiver backpressure** | When the receive ring is full, the frame is held & re-acked (never silently dropped); delivers when the app drains. |
| **Epoch resync on reboot** | A rebooted peer resyncs sequence numbers automatically (no manual reset). |
| **Session reset on host reconnect** | A new host session never inherits the previous one's stale queued/in-flight bytes. |

### Host (USB) link
| Capability | What it solves |
|---|---|
| **TinyUSB OTG-CDC stack** | Real USB flow control (NAK) instead of the ESP32-S3 USB-Serial/JTAG silently dropping bytes on bursts (the original ~270 B/s ceiling). |
| **Checksummed USB framing + length-cap resync** | Detects any corruption/desync on the USB hop and re-syncs within one frame ‚Äî never delivers garbage. |
| **PSRAM ingest ring + bulk USB reads** | A fast `cat bigfile` floods USB (~1 MB/s; the "115200 baud" is cosmetic) far faster than the ~2 KB/s link. The arduino-esp32 USB-CDC core silently drops on overflow (no NAK ‚Äî upstream bug [#10836](https://github.com/espressif/arduino-esp32/issues/10836)). We bulk-drain the FIFO into a 2 MB PSRAM ring ‚Üí every realistic transfer is absorbed losslessly with **no client-side flow control**. Verified: 128 KB fast-overrun transfer byte-exact (`idrop=0`). |
| **Always-drain `usbPoll` + 2 KB staging** | Never stalls the USB read, so the FIFO doesn't overflow; backpressure lives downstream. |
| **Non-blocking host TX ring** | A slow terminal can't block the radio loop (a blocking write used to wedge the whole link). |
| **XON/XOFF flow control** + **tiny bulk pacing** | Keeps the host from overrunning the link ring / USB FIFO at the fastest modes. |
| **No-reset serial open** | Opening the port doesn't toggle DTR/RTS into the bootloader (which bricked the link). |
| **Reconnect output flush** | Reconnecting a terminal doesn't get the previous session's buffered output dumped at it. |
| **USB-hop diagnostics** (`ugood`/`udrop` telemetry) | Makes host-hop byte loss measurable ‚Äî turned guesswork into root-cause. |

### Radio / RF
| Capability | What it solves |
|---|---|
| **Per-mode ToA-derived timing** | Turn timing auto-computed from `getTimeOnAir()` ‚Üí any SF/BW is tuned automatically, no hand-constants. |
| **Auto TX power control** (symmetric-link heuristic) | Backs power off when the link is over-strong (front-end saturation at close range) and ramps up when weak ‚Äî never above the configured legal max. Foundation for ADR. |
| **Configurable SF/BW/CR/power + named modes** | turbo / fast / medium / slow / far ‚Äî speed-vs-range presets over the standard knobs. |
| **Half-duplex turn-taking MAC** | Coordinates a single shared radio for two peers with minimal dead time. |

### Security & integrity
| Capability | What it solves |
|---|---|
| **Ascon-128 AEAD encryption** (pre-shared key) | Confidentiality **and integrity**: the NIST-lightweight-standard cipher (validated against the official KAT vectors). Every frame ‚Äî header included ‚Äî is authenticated, so tampering/bit-flips are detected and dropped. |
| **64-bit nonce counter persisted in NVS** | No keystream reuse across reboots: the counter resumes past its saved value (skips unsaved counters), the exact LoRaWAN-1.1 fix for the counter-reset bug. |
| **Sliding-window anti-replay** | Recorded ciphertext frames can't be re-injected within a session (64-frame window tolerates reordering). Plaintext frames are rejected on an encrypted link. |
| **Link watchdog** | Tears down the login session when the air link drops, so a reconnect must re-authenticate (no resumed authenticated shell). |
| **LoRa hardware CRC + per-frame integrity** | Corrupt air frames are dropped and retransmitted (never delivered). |

### Compression & data
| Capability | What it solves |
|---|---|
| **Per-frame heatshrink compression** (only if it shrinks) | Cuts airtime on compressible data (text/logs ~2√ó the raw rate); never expands incompressible data. |
| **Byte-exact transparent pipe** | Arbitrary binary passes intact end-to-end (verified with random payloads). |
| **Pure transparent byte pipe** | The link stays a clean byte pipe; any terminal cooking (pty/line discipline) is left to the OS at each end ‚Äî proper layering. |

### Usability
| Capability | What it solves |
|---|---|
| **Client = any plain serial terminal** | No special app on the roaming end ‚Äî tio, PuTTY, a phone serial app all work. |
| **AT/`+++` command mode** (Hayes-style) | Configure the radio (power/address) and query live link state (`AT+LINK?` ‚Üí RSSI/SNR/power) out-of-band from the *same* plain terminal ‚Äî no host software, no extra wires. Covers the link-state role DCD can't (TinyUSB CDC has no device‚Üíhost carrier API). |
| **Unified firmware** (role from address) | One binary for both ends; the lower address auto-initiates ‚Üí boards are interchangeable, no per-board build. |
| **OS-native login over LoRa** (`agetty`) | Because the link is a transparent serial port, you point `agetty`/`serial-getty` straight at it for a real PAM login + shell ‚Äî no project-specific server. |
| **Native simulation test suite (26 tests)** | Loss, reordering, backpressure, compression edges, reconnect resync, SR efficiency, AEAD KAT/replay, X25519 pairing ‚Äî caught real bugs before hardware. |
| **Throughput harness + flashing helper + udev rules** | Reproducible benchmarking; button-free flashing over native USB. |

## Problems we hit, and how we solved them (the journey)
A condensed engineering log ‚Äî each was a real bug or wall, and the fix became a feature above.

1. **ISR crash under load** ‚Äî `digitalWrite` in an IRAM ISR crashed the chip ‚Üí moved the LED out of the ISR.
2. **Decompress hang** ‚Äî heatshrink could loop forever ‚Üí output-bound guards.
3. **Bulk stall at ~240‚Äì850 B** ‚Äî the USB-Serial/JTAG stack dropped host bytes on bursts ‚Üí first paced, ultimately **switched to TinyUSB** (real flow control). ~270 ‚Üí 3000+ B/s.
4. **Compression corrupted incompressible data** ‚Äî a `c>255` length truncation; the sim missed it because tests only used *compressible* payloads ‚Üí fixed + added an incompressible test.
5. **Login froze after password** ‚Äî agetty's auto-baud mangled input ‚Üí spawn `login(1)` directly.
6. **Reconnect desync / resumed sessions** ‚Äî epoch handshake + a link watchdog that logs out on link loss.
7. **The phone-terminal saga** ‚Äî chased telnet IAC and CR/LF translation; the real cause was the USB stack. Removed the telnet/CR cruft (the pty handles CR; the byte ferry stays transparent).
8. **The multi-window stall on full frames** ‚Äî Go-Back-N + a tiny window + rigid turn timers ‚Üí **Selective-Repeat + SACK + BDP window + per-frame burst deadline**. Bulk binary became reliable.
9. **USB framing desync** ‚Äî a single dropped byte corrupted the stream forever ‚Üí **checksum + length-cap resync**.
10. **Cross-run stale data / dirty reconnects** ‚Äî the board kept link state across host sessions ‚Üí **session reset on reconnect**.
11. **RX overload vs USB-hop drops** ‚Äî strong-signal corruption *looked* like RF overload; `udrop` telemetry proved it was the **USB hop at high feed rate** (fixed with a tiny pace) while **auto TX power** handles real saturation.
12. **Crypto was confidentiality-only** ‚Äî AES-CTR had no integrity (bit-flipping), reused keystream on reboot (counter reset to 0), and no replay protection. Replaced with **Ascon-128 AEAD** (KAT-validated) + **NVS-persisted 64-bit counter** (reboot-safe nonce) + **anti-replay window**, header authenticated. Tag length is a build flag (8/16).
13. **Large transfers dropped/stalled at ~30‚Äì50 KB** ‚Äî *not* the radio or transport (both proven byte-exact at 256 KB / 1 MB in sim, and at 128 KB on hardware once paced). A per-frame byte counter (`hin`) pinpointed the loss to **host‚Üíboard USB ingestion**: the arduino-esp32 USB-CDC core drains the FIFO into a small queue and drops on overflow (no NAK ‚Äî upstream [#10836](https://github.com/espressif/arduino-esp32/issues/10836)) when a fast `cat` outruns the link. Fixed with a **2 MB PSRAM ingest ring + bulk reads**; 128 KB fast-overrun now byte-exact with zero drops.
14. **Turbo was at ~half its PHY ceiling** ‚Äî profiling showed only ~24% airtime used, so turbo was **turnaround/overhead-bound, not airtime-bound**. Two fixes ~**doubled it to ~8 KB/s** (‚âà 56K modem): a **BDP-sized per-mode window** (16 frames/burst at turbo so one turn-around amortizes over a long burst; auto-scaled from ToA, down to 2 at SF12), and **skipping the 30 ms interactive-echo piggyback wait on one-way bulk** (pure dead-time in the ACK path). Reliability unchanged (128 KB overrun + 24/24 sim still byte-exact).
15. **Auto-power starved the link dead (the symmetric-link assumption)** ‚Äî a responder went deaf mid-run (`rssi` pinned, `snr` garbage, `tx=0`) while the initiator hit ~100% retx with its power floored at `pwr=-10`. Root cause: `AdjustTxPower()` set *our* power from the RSSI *we* receive, assuming a symmetric link. On an asymmetric path (different loss each way + a noisy close-range bench) the side that hears its peer strongly floors its own power, which drops the peer below its demod SNR ‚Äî a stable, dead equilibrium. Modeled it in the sim (`test_autopower_own_rssi_starves_asym`) and it **reproduced the exact hardware numbers** (initiator floors to -10, responder can't demod, initiator still hears the responder). Fix: **auto-power is now opt-in (`FEAT_APWR`, `AT+APWR`), OFF by default** ‚Äî fixed power keeps an asymmetric link up (sim-validated); a correct peer-SNR-feedback loop (also sim-validated) is future work. This restored the dead link to byte-exact.
16. **No recovery for a wedged radio** ‚Äî a coredump confirmed the responder had panicked, and there was no firmware path to recover a radio whose RX silently wedges (rendezvous only changes *mode*, and only in `auto`). Added a **radio-stuck watchdog**: on prolonged no-valid-RX it does a full `begin()` re-init at the *current* mode (mode- and link-preserving, runs in all modes), layered below rendezvous. Validated in sim first (`test_sim_radio_stuck_*`, incl. a teeth test that the link stays dead without it). Caveat found on hardware: a full **reboot** clears a wedge that `begin()` alone does not ‚Äî a future watchdog should `reset()` the chip (toggle NRST) for a deeper recovery.
17. **"Encryption broke the speed?" ‚Äî ruled out by isolation** ‚Äî fast modes ran below the README numbers with ~19‚Äì33% retx *at +20 dB SNR* (loss that isn't SNR-driven). Hypothesis: the per-frame Ascon decrypt. Tested directly with `AT+ENC=0` on both ends: throughput was **unchanged** (ENC off ‚âà ENC on) ‚Äî decrypt is ~tens of ¬µs on a 240 MHz S3, far too fast to matter. The real cause is the **RX re-arm deaf window**: the receive path did `startReceive ‚Üí readData ‚Üí standby` *per frame* and re-armed on the next call, so a back-to-back frame arriving during inter-frame processing was missed (research confirms this is a known RadioLib/SX126x anti-pattern ‚Äî the SX126x stays in RX after a packet). Modeled it in the sim (`test_phys_rx_rearm_miss_collapses`: continuous-RX ~2.3√ó faster than per-frame standby). Implemented continuous RX (arm once, no per-frame standby) behind a build flag **`RX_CONTINUOUS` (default OFF)**. Along the way found a real bug: `startTransmit()` assumes the radio is in standby (RadioLib's blocking `transmit()` standbys first), so leaving continuous RX requires an explicit `standby()` before TX or the transmit is malformed ‚Äî fixed. It is **OFF by default and should not be enabled**: besides being unvalidatable on a bimodal bench (a degraded window made the known-good per-frame build fail identically), the continuous-RX build **HUNG the responder hard** ‚Äî zero LED activity, unresponsive to AT, **no panic/coredump, and the 25 s software watchdog did not recover it** (only a reflash did), a failure mode never seen on the per-frame build. So it's two problems: a continuous-RX trigger *and* a more general **watchdog gap** (a hung loop must always self-recover, and this one defeated the watchdog ‚Äî likely a hang with interrupts/timer blocked). One good-window run did show turbo +25% (3.95 vs 3.17 KB/s), so the *direction* is right, but it must not ship until the hang is root-caused and the watchdog made to catch it. (`getRxBufferStatus` is *not* the issue ‚Äî RadioLib `readData()` already reads at the rolling offset; the real bug found was that `startTransmit` assumes STANDBY, so leaving continuous RX needs an explicit `standby()` before TX.)
18. **Throughput is compression- and range-sensitive, not broken** ‚Äî the README's headline rates are **compressible** data; with compression working (`medium` zeros measured **2.65 KB/s, above the 2.0 README**), random data is the worst case and PHY-limited (`medium` ‚âà 1.1 KB/s raw). Also confirmed the **cm-range bench is bimodal/wedgy** ‚Äî the same mode swings 0 ‚Üî ~3 KB/s run to run because point-blank range causes front-end saturation + local EMI (the laptop/USB next to the initiator's antenna read a ~-32 to -48 dBm noise floor vs the responder's ~-69). A reboot clears wedges. Real headline numbers need **field-range** testing, not a desk bench.
19. **A hung loop defeated the software watchdog (robustness gap) ‚Äî fixed.** A deeply hung firmware state showed no LED, no AT, no panic, and the 25 s `esp_timer` watchdog never rebooted it. The fix: put the loop task under the IDF **Task Watchdog** (`enableLoopWDT()`; the framework has `CONFIG_ESP_TASK_WDT_PANIC=y`, so a stalled loop now reboots), plus a software **no-progress reboot** (`MaybeReboot`/`DiagRebootNoProgress`, `lastreset=NO-PROGRESS`) for a deaf-but-alive wedge, and an **NRST hardware radio reset** (`radio.reset()`) in the radio-stuck recovery. Recovery is now layered: radio re-init ‚Üí rendezvous ‚Üí NRST reset ‚Üí reboot ‚Üí TWDT. Validated: boards self-recover instead of needing a reflash.
20. **Continuous RX vs the SX1262 AGC lockup ‚Äî and a wrong turn corrected.** The throughput ceiling is the per-frame RX **re-arm deaf window** (~20-40 % retx at high SNR). Early continuous-RX attempts (arm-once `RX_CONTINUOUS`, then an **interrupt-driven** radio task) showed the responder go deaf (`snr~-27`) where per-frame stayed healthy (snr ~12), which pointed at the "AGC lockup in RX" errata (per-frame's standby-between-frames resets the AGC). The fix was *thought* to be `restartAGC()` "in a newer RadioLib." **Both halves of that were wrong** (corrected 2026-06-30): 7.7.1 *is* the latest RadioLib, and the method is `resetAGC()` and already present ‚Äî no upgrade needed. See entry 22 for what `resetAGC()` actually did.
21. **Close-range desense ‚Äî lower the power.** Much of the session's "deafness/wedge" at the cm bench was **front-end desense** from +22 dBm transmitters centimetres apart. Measured (per-frame medium): 22 dBm ‚Üí 0.39 KB/s / 44 % retx; 10 dBm ‚Üí 0.48 / 23 %; **2 dBm ‚Üí 0.69 / 28 %**. So fixed-max power is wrong at close range; reducing it recovers the link. The proper fix is a correct **peer-SNR-feedback auto-power** (#53) that lowers power when the peer hears us strongly ‚Äî which also cures the earlier asymmetric-starvation. (Neither Meshtastic nor Reticulum auto-adjusts power; this is our own feature.)
22. **`resetAGC()` WEDGES the SX1262 ‚Äî the real cause of the "responder hangs" saga.** To prevent the AGC lockup (entry 20) I added RadioLib's `resetAGC()` (warm `sleep(true)` ‚Üí recalibrate) to the continuous-RX path. After that build ran, **every** subsequent reflash ‚Äî including the known-good **per-frame** build ‚Äî left the responder dark (no LED/AT/coredump). Root-caused with no JTAG by adding a retry loop that prints the error: `radio.begin()` returns **err -2 `CHIP_NOT_FOUND`**, and even an aggressive NRST (3√ó100 ms) won't clear it ‚Äî **only a VBUS power-cycle does.** So `resetAGC()`'s `sleep()` left the chip unreachable over SPI, and the old `setup()` `while(true){delay(500)}` dead-radio trap then *looked* like an MCU hang on every boot. This also means the earlier continuous-RX/INTR_RX "deaf/hang" results were partly against an already-wedged radio. Fixes: (a) **removed `resetAGC()`** ‚Äî AGC reset uses a plain `standby()`; (b) reworked continuous RX to a **single-threaded main-loop** design (`RX_CONTINUOUS`); (c) replaced the silent dead-radio trap with `RadioHardNrst()` + a **retry/self-heal** bring-up so a wedged radio is visible and recovers on power-cycle instead of hanging dark; (d) added `-DDEBUG_BRINGUP=1` staged serial markers for no-JTAG bring-up debugging (lesson: native-USB CDC drops boot-time prints with no DTR, so only *continuously* printed markers are visible). Lesson: **never warm-`sleep()` this SX1262 in the hot path**, and never let `setup()` hang silently on a radio failure.
23. **The real villain all along: a loose board-to-board connector.** After the `resetAGC` wedge (entry 22) survived even a VBUS power-cycle, RadioLib debug showed the SX1262 version register reading a constant mostly-`0xFF` (`ff ff ff fe 0b ‚Ä¶`) ‚Äî the chip wasn't driving MISO. Not a software wedge: a **marginal B2B connector** on one XIAO/Wio-SX1262 unit (pressing/wiggling it made the LED resume). Swapping in another board fixed it instantly. **This intermittent connection ‚Äî not firmware ‚Äî was behind most of the multi-session "responder goes deaf / wedges / hangs" saga.** On a *solid* link the truth is simple: the **per-frame path is already 0 % retx** at fast modes (the earlier 42‚Äì58 % retx was the flaky connector + close-range desense), and **interrupt-driven RX (`INTR_RX`) is byte-exact across every mode** with no hang/deafness. Continuous RX gives ~no throughput gain over per-frame on a good link (both 0 % retx) ‚Äî its whole rationale (the re-arm deaf window) was really the hardware. Lessons baked in: a self-healing radio bring-up (entry 22c), and ‚Äî to make the *next* loose-connector instantly obvious ‚Äî moving device identity + the USB descriptor to the chip MAC so the `/dev/serial/by-id` name self-documents (planned). Throughput finally measured clean (INTR_RX, byte-exact, both compression cases) ‚Äî see THROUGHPUT.md.
24. **Slow modes need ToA-scaled recovery timeouts.** With a solid link, `slow` (SF9) worked but `far` (SF12) reboot-looped: both ends hit `lastreset=NO-PROGRESS`. The recovery watchdogs (radio-stuck reinit 5 s, rendezvous 9 s, no-progress reboot 45 s) were **fixed** while SF12 airtime is ~4√ó slower, so the 5 s radio-stuck reinit fired *mid-handshake* (a far turn is ~8 s), disrupting the link until the 45 s reboot. Fix: each timeout is now `max(floor, mult √ó g_retransmit_ms)` ‚Äî the ARQ retransmit timeout already scales with time-on-air ‚Äî so the watchdogs stay eager on fast modes (the proven 5/9/45 s floors) and patient on slow ones, preserving the order stuck < rendezvous < reboot. (Also fixed a real `INTR_RX` bug found here: the radio task's idle re-arm timeout aborted slow/far frames whose ToA exceeded it ‚Äî it now blocks on `portMAX_DELAY` and re-arms only after `RX_DONE`.)
25. **Channels: investigated and removed.** We briefly shipped a global numbered-channel scheme (`AT+CHAN`) ‚Äî one integer grid laid over the SX1262's whole 150‚Äì960 MHz range, `freq = 150 + chan √ó bandwidth`. In use it was a poor fit: the numbers were huge and unintuitive (US 915 MHz ‚âà channel **6120** at BW125), and worse, **bandwidth-dependent** ‚Äî because the grid step *is* the current bandwidth, the same frequency is a *different* channel at every mode, so a channel number meant nothing without also stating the mode. Channels only really earn their keep as a small, fixed, **per-region** plan (LoRaWAN-style EU868 ch0‚Äì7, US915 ch0‚Äì63). Rather than keep a confusing universal grid, we **removed `AT+CHAN`/`AT+CHAN?`** and made `AT+FREQ` (plain MHz, mode-independent) the single way to set the carrier ‚Äî folding the old grid's range check into `AT+FREQ` (rejects outside ~150‚Äì960 MHz) so nothing was lost. Region-aware *named* channel plans may return alongside the FHSS work on the [roadmap](./ROADMAP.md), where a shared channel list is the natural foundation.
26. **far/SF12 delivered nothing ‚Äî a TX-safety timeout shorter than the airtime.** On a solid link, every mode worked except `far`: data was ingested but the responder's `hout` stayed 0 with ~75% retx and a radio-stuck `reinit`, identically at `pwr=22` and `pwr=2` (so *not* desense ‚Äî ruling that out at the 1.5 m bench took a low-power re-run). Root cause: `TxFrame()`'s stuck-radio safety timeout was a **fixed 8 s**, but a far/SF12/BW125/CR4-8 frame's time-on-air is **~13 s** ‚Äî so the TX wait broke early and `finishTransmit()` ran on a **truncated frame** the peer could never decode. The comment literally said "Longer than any real ToA," true until SF12 was added. Fixes (sim-first): (a) the safety timeout now **scales with the frame ToA** (`2√óg_toa_ms + 1 s`, floored at 8 s) so it always outlasts the airtime ‚Äî safe because `DiagPet()` (the RX idle hook) feeds both watchdogs throughout the long wait (`test_tx_safety_exceeds_toa_every_mode`); (b) the ARQ retransmit timer became **window-aware** (`(window+2)√óToA√ó1.4`, was a fixed `12√óToA` ‚âà 108 s at far) so a lost tail/lone frame ‚Äî which the SACK fast-retransmit can't see ‚Äî recovers in ~one round-trip, not ~2 minutes (`test_far_tail_loss_recovers_promptly`, overhead 14.2‚Üí6.1 ToA); (c) the `AT+SPEEDTEST` deadline now **scales its per-byte budget by ToA** (the old fixed 20 s + 4 ms/byte falsely reported a far TIMEOUT). Result on hardware: far now transfers **byte-exact at 0 % retx** (1 KB delivered, responder `hout=1024`). Lesson: every fixed timeout must be checked against SF12's seconds-long airtime ‚Äî the same class of bug as entry 24's recovery watchdogs.
27. **First-boot proximity pairing (zero-touch unique keys).** MAC role election already made the two boards interchangeable, and `AT+TRAIN` already derived a unique per-pair key over X25519 ‚Äî but training was a manual step, and without it a pair fell back to the shared built-in key (no per-pair secrecy). Proximity pairing closes that: a board with **no key in NVS** boots straight into pairing ‚Äî drops to **low TX power** so it reaches a *nearby* board (proximity = human intent, like NFC/BLE), discovers + elects roles, auto-runs the `AT+TRAIN` exchange, **persists role + key**, then jumps to full power; every later boot loads that identity and **skips discovery**. Surfaced for the operator: a pairing LED blink, a periodic `[LoRa-Serial] PAIRING ‚Ä¶` line on the USB port, and `ATI state=pairing`. Re-pair with `AT+PAIR` (on demand) or `AT&F` (wipe ‚Üí re-pair next boot). Built behind `PROX_PAIR` (default ON after validation; `-D PROX_PAIR=0` restores re-elect-every-boot). Hardware-validated on two boards: boot-pairing from an empty NVS (fresh key, both ends matching, persisted), `AT+PAIR` re-pair, paired-boot-skip, and a working data link after pairing ‚Äî all confirmed. **Known gap:** low power alone is a weak proximity gate ‚Äî ~0 dBm still linked cleanly at 1.5 m on the bench (SNR 13), so adjacency isn't strictly enforced yet; an RSSI gate (accept a pairing beacon only above, say, ‚àí45 dBm ‚Äî ‚àí30 at 5 cm vs ‚àí56 at 1.5 m cleanly separates them) is the planned hardening.
28. **Full-auto 1 MB wedge ‚Äî a watchdog-starvation bug (fixed) layered over an SX1262 mode-switch deaf-lockup (characterized).** With ADR (`auto`), peer-SNR auto-power, and the GFSK rung all on, a 1 MB transfer reboot-wedged at ~18 s every time, killing the host transfer (USB drop). The breakthrough was a **non-perturbing RTC breadcrumb**: each blocking radio op the loop runs stamps `g_dbg_stage`, the RX task stamps `g_dbg_rx`, both `RTC_NOINIT` bytes snapshotted in the IDF Task-WDT ISR *and* the no-progress path so they survive the reboot, reported by `AT+DIAG` as `wedgeop`/`rxop`. (Earlier *serial* markers Heisenbug-shifted the hard wedge into a soft one ‚Äî a byte write to RTC doesn't perturb timing.) It exposed two independent problems. **Layer 1 ‚Äî watchdog starvation during recovery (FIXED).** When the link went deaf, the recovery path itself ran >5 s without feeding the 5 s task watchdog, so *recovery tripped the very watchdog meant to let it run* ‚Üí instant reboot that dropped the transfer. The unfed stretches: `Radio::Reinit()`'s aggressive NRST (~0.9 s of `delay()`) + `begin()` (each SPI op can burn the full BUSY timeout on a wedged chip); the `Rx()`/`Tx()` wait loops whenever the host-I/O idle hook was disabled (pairing NULLs it) or `Rx()` returned at once on a ready-frame burst (it fed only *after* the `Pop`); and `Host::Poll()`'s USB-ingest / `IngestToLink` loops under a host flood. Fix: feed `feedLoopWDT()` through all of them. Result: the task-WDT reboot is **gone** ‚Äî the transfer now survives the ADR climb (~60 s vs 18 s) and only the *designed* no-progress recovery fires. **Layer 2 ‚Äî the SX1262 deaf-lockup on a switch under load (CHARACTERIZED).** With the watchdog fed, both ends now reach `lastreset=NO-PROGRESS, wedgeop=i` (in `Reinit`), `rxop=.` (RX task idle): the loop is *alive and recovering*, but an ADR mode switch under sustained bulk intermittently drives the radio into a deep deaf state that runtime recovery ‚Äî aggressive-NRST `Reinit` **and** rendezvous-to-medium ‚Äî does **not** clear; only a full MCU reboot does (consistent with the SX126x AGC/state errata, entries 20/22). It is **not turbo-specific** (capping the auto-climb at `fast` still deafed at ~58 s) ‚Äî it's the *switch under load* itself. By contrast a **fixed** mode is rock-solid for bulk: fixed turbo carried 1 MB **byte-exact at 3.91 KB/s with no reboot** (261 s, past the deaf window), 256 KB earlier the same. Real bugs fixed en route: auto-power *fights* ADR (it floors power for the current slow mode, then ADR climbs into a power-hungry one the now-too-low power can't sustain ‚Üí power was traced to ‚àí6 dBm right before a deaf switch; auto-power is now disabled under ADR); the runtime `Reinit` now uses the boot bring-up's aggressive NRST (a single `radio.reset()` couldn't clear the deep wedge); the RadioLib **SPI BUSY timeout was bounded 1000 ‚Üí 100 ms** (an unbounded op was a separate starvation vector) and the heap-`new`'d `Module` became a named global to reach it (rule 5); a high-priority RX-task **storm-breaker/yield** (so a noise-driven `RX_DONE` flood can't starve the loop); and the GFSK rung **goes deaf on the switch into it** (`wedgeop=f`) so it stays manual-only in auto. **How to use it:** pin a fixed mode (`AT+FMODE=turbo`/`fast`) for reliable sustained bulk ‚Äî the stable default path; ADR (`auto`) adapts the mode to changing range/SNR and suits interactive/intermittent use, but ADR + the GFSK rung remain **opt-in/experimental for sustained bulk** pending a fix for the switch-under-load deaf-lockup (the open lead: a recovery that re-inits the SPI peripheral + radio task *without* a full reboot, since only the MCU reset currently clears it). The `wedgeop`/`rxop` breadcrumb stays as a permanent `AT+DIAG` diagnostic.

29. **GFSK `ludicrous` went deaf ‚Äî the entry-28 storm-breaker throttle broke it (fixed).** After the interrupt-RX/OOP refactor, switching to `ludicrous` and sending data delivered **nothing** and reboot-wedged both boards ‚Äî even a *manual* `AT+FMODE=ludicrous` on both ends, so it wasn't the mode-switch handshake. LoRa was flawless, which localized the fault to the GFSK-specific path. The chase ruled out ‚Äî by hardware measurement + transcript forensics ‚Äî everything "obvious": the GFSK PHY config (bitrate/dev/RxBw/preamble/sync/CRC) was **byte-identical** to the last byte-exact `ludicrous` build, RadioLib was pinned unchanged, frequency/sync matched, and the low-level radio worked. Temporary `AT+LINK?` counters proved DIO1 **fires** for FSK (`rxd` climbs), `readData` **succeeds** (`rxf‚âà0`), and `startTransmit` **succeeds** (`txa==txo`) ‚Äî yet the received frames had **garbage lengths** (`getPacketLength` returned 0/178, only rarely a real 26): the receiver was demodulating noise and missing real frames. A forensic diff of the `RadioTask` drain loop against the last known-good checkpoint (which had hit `ludicrous 64 KB @ 9.83 KB/s byte-exact`) found the single regression: the **storm-breaker added in entry 28** appended a `vTaskDelay(1)` **after every processed RX_DONE**. Harmless to LoRa (RX_DONE is sparse; the tick is absorbed in the inter-frame gap), but **fatal to GFSK** ‚Äî its no-processing-gain receiver fires RX_DONE almost every wake, so the per-frame yield inserted a scheduling gap *between back-to-back burst frames*; the task could no longer drain + re-arm ahead of the next frame, so real frames were dropped/misframed ‚Üí deaf ‚Üí NO-PROGRESS reboot. The code was mistaking ordinary GFSK noise for a wedged radio. **Fix:** keep the FIFO-drain hot path **delay-free** (the proven design), and confine loop-starvation protection to a *genuine* deaf flood ‚Äî a successful frame resets the fail counter, so ordinary GFSK noise (which mostly passes CRC) never trips it; only many *consecutive* failed reads briefly halt RX for the loop's recovery window. **Results ‚Äî the same throttle broke *both* fast back-to-back-burst rungs (GFSK `ludicrous` AND LoRa `turbo`), so removing it fixed both:** **ADR (`AT+MODE=auto`) is fixed** ‚Äî it now climbs medium‚Üíturbo cleanly and carries **128 KB sustained at 4.83 KB/s, 0 % retx, zero reboots** (26 s, decisively past the ~17 s wall). That means **entry-28's "SX1262 deaf-lock under sustained bulk" was NOT a hardware errata ‚Äî it was this very throttle**, which had been added *while chasing entry 28*; the "fix" was the bug. **GFSK `ludicrous` is fixed in code** and carried 8 KB byte-exact at 5.91 KB/s when the RF was favorable ‚Äî but the **cm-range bench is bimodal for GFSK** (entry 18): at a slightly worse board position the demod yields garbage frames, an RF/bench limit (not the code) that needs field-range validation. Lesson (cf. entry 28): a throttle or timeout that is invisible on LoRa's chirp-correlated PHY can be lethal on the fast, sensitive, back-to-back-burst PHYs (GFSK *and* `turbo`) ‚Äî never put a per-frame delay in the RX drain path.

30. **Auto-power re-enabled under ADR ‚Äî the "conflict" was the entry-29 throttle; what's left is a power/throughput *tradeoff*, not a wedge.** Entry 28 had *disabled* peer-SNR auto-power whenever ADR was on, on the theory that the two fight: auto-power floors TX power for the current slow mode, then ADR climbs to a power-hungrier one the now-too-low power can't sustain ‚Üí deaf on the switch (power was traced to ‚àí6 dBm right before a deaf climb). But that "deaf" was mostly the storm-breaker throttle (entry 29); with the throttle gone, the pair was worth re-testing together. **They coexist cleanly.** Full `auto` + the GFSK rung + auto-power carried **1 MB end-to-end at 4.83 KB/s, 0 % retx, zero reboots** (212 s continuous), with auto-power actively trimming TX power to **‚àí10 dBm** at 1.5 m ‚Äî no deaf, no wedge. The old hard conflict was an artifact of the throttle bug, so auto-power now runs under ADR by default (a self-tuning encrypted link out of the box). **What remains is a genuine policy tradeoff, not a failure:** auto-power *minimizes* power, and at close range ‚àí10 dBm leaves ADR too little SNR headroom to climb to `ludicrous`, so ADR settles on `turbo` ‚Äî the same bench that reached ~10 KB/s on `ludicrous` with auto-power *off* runs ~4.8 KB/s with it *on*. Both are lossless; auto-power is simply optimizing for power, not throughput. The cost appears only up close (far away the signal is weak, so auto-power won't cut power and there is nothing to give up) and it protects the RX from close-range saturation. The open follow-up is **coordinated headroom** ‚Äî have auto-power target the current mode's demod floor *plus a rung of margin* so ADR can still climb, winning both low power and full throughput (to be modelled in the native sim first).

31. **Coordinated auto-power headroom ‚Äî auto-power now lets ADR climb instead of pinning it a rung low.** Entry 30 left this as the open follow-up: auto-power *minimizes* power for the CURRENT mode, so under ADR it held the peer's SNR just above the current rung's floor and never cleared the *next* rung's floor + margin ‚Äî so ADR could never climb, parking a rung low. The fix is `link_layer::AutoPowerTargetFloor` (shared verbatim by firmware and sim ‚Äî one source of truth): when ADR is driving, aim a rung *ahead* ‚Äî target the next-faster LoRa rung's floor, so the loop holds enough headroom for ADR to step up, then ratchets power up rung by rung until the link tops out (low power AND full throughput, not one traded for the other); a pinned mode still minimizes for itself. The firmware side is a one-line accessor, `Radio::next_faster_snr_floor()`, plus a conditional in `AdjustTxPower`. **The sim caught a real bug the instant the test ran:** the accessor first rounded the next floor with `lroundf` (like `snr_floor()`), but `lroundf(-12.5) = -13`, so auto-power held SNR at `-13 + 6 = -7` while ADR climbs off the *exact* float floor `-12.5 + 6 = -6.5` ‚Äî a ~1 dB under-provision that meant ADR *still* never climbed. Rounding the next floor UP with `ceilf` keeps the held margin above the climb threshold; the new closed-loop sim test (ADR picks the mode, auto-power sets the power, on a fixed channel) fails without `ceilf` and passes with it ‚Äî real teeth. **Scope:** the LoRa ladder only. The GFSK `ludicrous` top rung is gated by RSSI (‚â• ‚àí70 dBm), not SNR, so auto-power lowering power at close range drops RSSI below the gate and suppresses that climb ‚Äî a separate power-vs-RSSI tension left opt-in (entry 30). Bench: the fix is a no-op at the fastest LoRa rung, so at 1.5 m (where the strong signal already carries turbo) full auto + auto-power ran exactly as before ‚Äî 128 KB byte-clean at 4.83 KB/s, 0 % retx, zero reboots ‚Äî confirming no regression; the climb benefit shows at medium range, which the sim models and a point-blank bench can't reproduce.

32. **Auto-power starvation on reboot ‚Äî the quiet side couldn't re-establish.** Power-cycling the initiator (the "travel" board) left the link dead even though both ends were on `medium` with the same key ‚Äî *not* a mode/key/epoch problem. On the bench both boards showed it: the responder heard the initiator loud and clear (`snr=11.5`), but the initiator heard **nothing** back (`snr=0`, `hout=0`, 97 % retx, 31 radio re-inits). The tell was the power ‚Äî the responder sat at **‚àí10 dBm**. During the prior good session its peer-SNR auto-power had eased TX power all the way down (it heard the initiator strongly, so it *should* be quiet); after the initiator rebooted, that ‚àí10 dBm signal arrived below the initiator's ~‚àí80 dBm local (USB/EMI) noise floor, so the initiator couldn't hear it ‚Äî and the responder couldn't *learn* to boost, because the SNR feedback it needs rides the very reverse channel that was dead. A deadlock: the quiet side can't get louder without feedback, and feedback can't arrive until it's louder. Confirmed live by bumping the responder to full power ‚Äî the link re-established instantly (initiator `snr` 0 ‚Üí 12.5, both `retx` counters froze). **Fix ‚Äî two halves that close the broken feedback loop.** (1) *Honest reporting:* the deaf side stops sending a stale "I hear you fine" telemetry byte ‚Äî after `kApSilentMs` (3 s, above the idle turn cadence, below the 5 s radio-stuck floor) with no valid RX it reports the sentinel `kApNoSignalSnr` on its frames (carried even on a poll, header byte `out[9]`), so the quiet peer's `AutoPowerStep` boosts to reach it; without this the quiet side never learns to get louder. (2) *Own-power ramp:* a `link_silent` node also ramps its **own** power UP (a new `link_silent` arg to `AutoPowerStep`) rather than easing DOWN on the stale report ‚Äî otherwise the deaf side drifts to the floor too and both starve. Together: the deaf (usually rebooted) side stays loud and signals "I can't hear you," the quiet side boosts until heard, then both settle. `kApNoSignalSnr` lives in `autopower.h` (shared with the sim); `kApSilentMs` in `fw_config.h`. **Teeth:** `test_autopower_deaf_side_reports_no_signal` ‚Äî a stale "fine" report leaves the quiet side starved at the floor, while the honest `kApNoSignalSnr` report drives it up to re-link; plus `test_autopower_recovers_from_silence` for the own-power ramp. The bug is symmetric (either side can starve the other after *its* reboot); in the usual "travel the initiator" setup the initiator reboots and reports no-signal, and the stationary responder boosts.

33. **Device‚Üíhost went one-way on some USB hosts ‚Äî a CDC IN pipe stuck by writing before the terminal opened.** A prior change (commit `bd7606d`, "deliver device‚Üíhost without requiring DTR") had swapped `HostTxDrain` from the Arduino `Serial.write` path to raw `tud_cdc_write`, to serve hosts that read without asserting DTR. It backfired: some hosts ‚Äî notably **Termius on Android** ‚Äî connected and wrote fine (host‚Üídevice worked) but **read nothing** back. A long chase ruled out, by isolation, the USB **descriptor identity** (product/manufacturer strings and VID/PID ‚Äî reverting to the old-build defaults changed nothing; the VID is variant-locked to Seeed `2886` regardless) and the arduino-esp32 **core version** (constant throughout ‚Äî its install date was a coincidence). The breakthrough was a **flash-persisted USB probe** (`env:usbprobe`, `src/usb_probe.cpp`): it logs a per-second timeline (DTR/RTS, baud, connect count, and crucially `wav` = CDC IN-FIFO free bytes and `wfail` = writes that couldn't complete) to NVS, so it survives the unplug and dumps on the next boot ‚Äî the only way to see what a phone-side board does. It showed the IN FIFO **saturating (`wav=0`) and never draining**: the host was issuing no reads. Research pinned the mechanism ‚Äî **TinyUSB [#2572](https://github.com/hathach/tinyusb/discussions/2572)**: writing to the CDC IN endpoint *before the terminal is open* leaves the IN pipe "stuck busy/claimed," after which the host reads nothing; the fix is to gate on `tud_cdc_connected()` (DTR). The ungated `tud_cdc_write` did exactly that forbidden thing (it wrote from boot regardless of DTR); the old `Serial.write` was safe precisely *because* it is DTR-gated. **Fix: gate `tud_cdc_write` on `tud_cdc_connected()`** ‚Äî never touch the IN pipe until a terminal opens it. This is the *exact* path the probe validated reliable (Termius Android/Linux, PuTTY, Android Serial USB app; reconnects resume in 1-2 s right where they left off). **A wrong turn worth recording:** we first reverted `HostTxDrain` to the old `Serial.write` (also DTR-gated) *assuming* it was equivalent ‚Äî but it was never what we validated, and on hardware it reconnected slower and flakier (5-20 s blanks) than the gated `tud_cdc_write`. Lesson: ship *exactly* what you tested, not something you assume equals it. To *rule out* a link-reset cause (`CheckHostReconnect` firing on DTR flaps) we also added a **peer-reset counter** (`preset` in `AT+LINK?`, counting peer epoch changes); a live phone test ‚Äî initiator on the phone, responder driven from the PC ‚Äî showed `preset=0` throughout, confirming the initiator was *not* resetting its link and the output path was the culprit. **A reboot red herring on the way:** we suspected the arduino-esp32 **DTR/RTS‚Üíbootloader auto-reset** (`enableReboot`, on by default) and briefly disabled it ‚Äî but the old firmware never disabled it and worked, and the probe (reboot left on) ran full sessions with no reboot: a terminal asserts DTR+RTS together and does **not** walk the specific 4-step sequence that fires the bootloader (only esptool's reset does). So `enableReboot(false)` was **unnecessary and dropped** ‚Äî leaving reboot on is what lets `tools/upload_flash.sh` reset the board into the bootloader button-free. (Flashing footnote: esptool's *default* auto-reset fails on the XIAO's native USB ‚Äî the reboot re-enumerates the port mid-connect ‚Äî so `upload_flash.sh` does the reset itself, a 1200-baud touch plus a DTR/RTS reboot-walk, then flashes the stable download port; `make flash` uses it, and `pio run -t upload` should not be used here.) **Client compatibility, hardware-validated:** Termius/**Android** ‚úÖ, Termius/**Linux** ‚úÖ, PuTTY/**Windows** ‚úÖ, the Android "Serial USB Terminal" app ‚úÖ. **Termius/Windows** ‚ùå is the lone holdout ‚Äî the probe log shows it **flaps DTR/RTS constantly** (‚âà60 line-state changes and 16 connect/disconnect events in 60 s) and never holds DTR steady like every working client, so our required DTR gate stays mostly shut. That is a Termius-Windows client-side quirk (needs further investigation), not a device fault; the workaround is PuTTY on Windows.

34. **Field testing: `auto` ratchets down to `far` and can't climb back ‚Äî the fix is safe defaults, not more auto.** In real (non-bench) use with the beta defaults (ADR + peer-SNR auto-power both on), the link kept **settling on `far` (SF12) and staying there** ‚Äî the slowest, longest-range rung ‚Äî and pinning a faster mode by hand didn't reliably stick. **Leading diagnosis ‚Äî an auto-power √ó mode-switch transient the close-range bench never hit.** Auto-power (`AutoPowerStep`, `kAutoPwrStepDb = 2` dB/tick) trims each side's TX power down to just a margin above the *current* mode's demod floor. But applying a mode switch re-uses the **held** TX power: `Radio::ApplyMode` leaves power untouched on a LoRa‚ÜíLoRa reconfigure and passes the saved `tx_power_` on the GFSK‚ÜíLoRa path ‚Äî it never *raises* power for a faster rung, whose demod floor is higher. So the instant ADR steps up, the faster PHY is **born below its floor ‚Üí silent**, and the mode-switch **probation timer (`probation_ms = 4000`)** reverts the unconfirmed PHY after 4 s ‚Äî long before auto-power's 2-dB-per-tick ramp can raise power enough to make the faster rung audible. Net effect: every attempted climb is stillborn and reverts, so the pair ratchets *down* to the most robust mode and parks there. Entry 31's `AutoPowerTargetFloor` (hold a rung of headroom so ADR *can* climb) fixes the **steady state** but not this **switch-instant transient** ‚Äî at the moment the PHY changes, power is still the old floored value. **Why entries 30‚Äì31 missed it:** they validated at a ~1.5 m bench, where the signal is strong enough to carry the fast rung outright (nothing to climb, floored power still ample); the deadlock only bites at **marginal field range**, where the fast mode's floor sits near the received SNR. The native sim models per-mode demod floors, but its closed-loop ADR+auto-power test ran a fixed channel that didn't exercise the born-deaf-on-switch case. **Decision ‚Äî ship the reliable path by default.** The factory default is now a **fixed `medium` mode at full TX power** (`FEAT_DEFAULT` = `FEAT_COMP | FEAT_ENC`); ADR (`AT+MODE=auto`) and auto-power (`AT+APWR=1`) are **opt-in and documented experimental**. Rationale: fixed-mode + full power is rock-solid (fixed-mode bulk is byte-exact at 0 % retx ‚Äî entries 23/28), whereas the adaptive layers ‚Äî though implemented and sim-green ‚Äî have an unproven field envelope. This is a beta owning its limits, not a regression: the features stay in, one command away, clearly marked as needing more validation. **Deferred fix (future work):** carry power *across* a faster switch ‚Äî when ADR proposes a higher rung, raise TX power to that rung's floor + margin *as part of* applying the switch (a switch-time helper in `autopower.h`, shared verbatim firmware‚Üîsim, √Ý la `AutoPowerTargetFloor`), so the new PHY is born audible; optionally lengthen probation for an upward step. Model it in the native sim first with a **marginal-SNR** closed-loop test that reproduces the born-deaf revert (the teeth entry 31's fixed-channel test lacked), then field-validate before it returns to on-by-default. Docs updated to the new default: `README.md` (defaults + an experimental note), `docs/DESIGN.md` (¬ß5 shipped-default), and `src/fw_config.h` (`FEAT_DEFAULT` + the feature-flag comments).

35. **MAC role auto-election was silently compiled out ‚Äî a lost `#define` in the refactor, masked by persisted NVS.** Setting the out-of-box defaults (entry 34) meant `AT&F`-ing both bench boards to a clean NVS ‚Äî and the pair came up **dead**: both boards reported `addr=1 peer=0 initiator=1`. Two initiators is structurally impossible when role election runs ‚Äî `ElectInitiator` makes the numerically-lower MAC the sole initiator ‚Äî so the tell was unmistakable: **the election wasn't running at all.** Root cause: the whole MAC-role-discovery + proximity-pairing subsystem is guarded by `#if MAC_ROLE` / `#if PROX_PAIR`, and **nothing defined those macros.** In the pre-release monolith they defaulted on via `#ifndef MAC_ROLE / #define MAC_ROLE 1` (and the same for `PROX_PAIR`) sitting in `main.cpp` right above the discovery code. The static-singleton refactor moved the *code* into `fw_device.cpp` (still `#if MAC_ROLE`) and even pointed `fw_device.h` at the intended new home ‚Äî `#include "fw_config.h"  // MAC_ROLE / PROX_PAIR build toggles` ‚Äî but the two `#define ‚Ä¶ 1` **defaults were never re-added there.** An undefined macro is `0` in `#if`, so `DiscoverRole()`/`ProximityPair()` compiled to nothing; `RecomputeRole()` then fell back to the static `NODE_ADDR` default (1) on **both** boards ‚Üí both initiator, no link (and no proximity pairing, so the pair also silently stayed on the shared built-in key instead of deriving a unique one). **Why it stayed hidden through the public release:** NVS survives a reflash, and the bench/field boards already had **distinct roles persisted** from pre-refactor builds ‚Äî `LoadSettings()` re-read `addr=1/peer=2` and `addr=2/peer=1`, so the link kept working and nobody re-elected. Only wiping NVS (`AT&F`, or a genuinely fresh board) exposed the dead default. **Why the tests didn't catch it:** the native suite includes `test_identity_election`, but it exercises `identity.h` **directly** ‚Äî that pure logic was always correct; the bug was the firmware not *compiling it in*. Native builds set `build_src_filter = -<*>` (they don't build `src/` at all), so no native test can observe a `src/`-level `#if` compiling a feature out. **Fixes.** (a) Restore the defaults at the include's documented home: `#ifndef MAC_ROLE / #define MAC_ROLE 1` and `#ifndef PROX_PAIR / #define PROX_PAIR 1` in `fw_config.h`, so the single `node_raw` image auto-elects and pairs on both boards again. (b) A **build-time guard** in `fw_device.cpp` for the half a native test can't reach ‚Äî `#if !MAC_ROLE && (NODE_ADDR == 1) && (PEER_ADDR == 0) ‚Üí #error`: the exact "both boards default to address 1" regression is now a compile failure, while a deliberate legacy static-address build (distinct `NODE_ADDR`/`PEER_ADDR` per board) still compiles. (c) A **discovery-outcome test** with teeth (`test_mac_discovery_outcome`): it simulates *both* boards' views of a MAC exchange and asserts the safety invariant over the real bench MACs, a near-tie pair, and a 2000-pair random sweep ‚Äî exactly one initiator, mutually-agreed addresses, always distinct (nonce-safe) ‚Äî so "both initiator" can never originate in the election logic. **Hardware-validated after the fix:** fresh `AT&F` on both, then they auto-elected complementary roles (lower MAC ‚Üí `initiator=1 addr=1 peer=2`; higher ‚Üí `initiator=0 addr=2 peer=1`), proximity-paired to a **unique** shared key, and carried a speedtest byte-exact at **0 % retx**. Lesson: a feature toggled by a `src/`-level `#define` has an **enablement** state that unit tests (which bind to the portable logic, not the firmware build) can't see ‚Äî guard the enablement at build time, and don't let persisted state mask a fresh-boot regression.

36. **A display node with nobody reading its USB throttled itself to a dead stop ‚Äî DTR back-pressure with no reader wedged the whole link.** On the Wio Tracker L1 (a receive-and-display node ‚Äî you watch the OLED, nothing has its USB serial port open), a stream from the peer would flow for a moment and then **freeze**: the heartbeat kept pulsing and the signal meter stayed up (so the link *looked* alive), but no new text reached the screen and the byte count stopped. The dead-giveaway number: the L1 stalled at **`hout=2010`**, and the device‚Üíhost ring is exactly **`kHostTxRingBytes = 2048`** ‚Äî it filled once and never drained. **Root cause ‚Äî the DTR gate from entry 33 starves a reader-less display node.** `HostTxDrain()` only writes to the USB CDC IN endpoint when a terminal has the port open (`platform::HostCdcConnected()` == DTR asserted) ‚Äî the invariant entry 33 established to avoid wedging the pipe. But on a display node DTR is *never* asserted, so `HostTxDrain` returned immediately every pass and the 2048-byte `host_tx_` ring never emptied. Once full, `PumpLinkOut()` stops reading from the link (`if (room <= kHostRingHeadroom) break;` ‚Äî deliberate receive back-pressure), the link's receive window fills, ARQ back-pressures the **peer**, and the sender stalls ‚Äî while control/ACK frames still flow, so the heartbeat and RSSI stay live and mask the stall. Worse, the OLED teletype is fed from that *same* ring (`HtPush` mirrors each delivered byte to `g_host_out_hook`), and `HtPush` only runs when `PumpLinkOut` delivers ‚Äî so the screen freezes in lock-step with the link. Net: a node whose entire job is to display received data throttles itself dead after one 2 KB ring-full, because nothing is draining a USB port that, on a display node, *nobody will ever open.* **Fix ‚Äî discard, don't hold, when there's no reader.** Buffering device‚Üíhost output while DTR is deasserted is pointless: `CheckHostReconnect()` already drops the whole backlog the instant a terminal (re)connects (a fresh session), so those bytes are never delivered anyway ‚Äî holding them only wedges the link. So when `!HostCdcConnected()`, `HostTxDrain` now **discards the ring** (`ht_tail_ = ht_head_`) instead of returning with it full. `HtFree()` stays high, `PumpLinkOut` keeps pulling from the link at line rate, `HtPush` keeps feeding the OLED, and the link never stalls ‚Äî the display updates continuously with nothing attached to the serial port. The CDC-pipe safety invariant is untouched (we still never touch the IN endpoint before DTR); we just stop letting an undrained ring back-pressure the radio. **Why the native sim couldn't catch it:** this is `src/fw_host.cpp` glue + real USB-CDC/DTR behavior, and native builds set `build_src_filter = -<*>` (they don't compile `src/`), so no native test can observe it ‚Äî a two-board-bench bug by nature (CLAUDE.md rule 6). **Hardware-validated (XIAO‚ÜíL1, nothing reading the L1's USB):** 5.6 KB streamed ‚Üí `hout=5601` where the old firmware capped at ~2010; then a **1 MB** sustained transfer ran lossless ‚Äî `idrop=0` throughout, the OLED showing **strictly sequential** line numbers (‚Ä¶9002, 9003, 9004‚Ä¶ with no gaps, i.e. byte-perfect in-order delivery), the ring cycling ~500 times with no re-wedge. Lesson: a safety gate that's correct for the *interactive* case (don't write to a pipe no terminal has opened) can be a **liveness bug** for a *headless* one (a display node has no terminal, ever) ‚Äî when you gate output on a reader, decide what happens to the data when there is no reader, and never let that decision back-pressure something upstream that can't afford to stall.

## Where this stands vs Reticulum ‚Äî for point-to-point
We are **not** a mesh and don't try to be. For the narrow *two-node reliable serial*
job, the specialization wins on several axes:

- **Leaner MAC:** deterministic `F_MORE`/burst hand-off vs CSMA carrier-sense + backoff (pure overhead with only two nodes).
- **No networking tax:** no addresses, routes, or per-packet routing headers.
- **Link-tuned reliability:** SACK + a window sized to *this* link's bandwidth-delay product, timing derived from its exact time-on-air.
- **Auto power control** built in (opt-in/experimental ‚Äî overload protection + energy), without a network server.
